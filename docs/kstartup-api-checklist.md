# K-Startup API 연동 전 체크리스트

**중요**: 현재 서비스 중인 **"창업진흥원_K-Startup(사업소개,사업공고,콘텐츠
등)_조회서비스"** 기준. 폐기된 **"창업진흥원_창업지원공고(K-Startup)"** API
기준으로 작성하지 않는다.

공식 문서/Swagger에서 확인되지 않은 endpoint·파라미터명·응답 필드키는
추측해서 채우지 않고 "확인 필요"로 남긴다.

## 2. 상세 기능 목록

공식 조회 샘플 페이지 기준 4개 하위 서비스 확인됨:
- 통합공고 지원 사업 정보
- 창업관련 콘텐츠 정보
- 창업관련 통계보고서 정보
- **지원 사업 공고 정보** ← 우리 앱이 우선 연동할 대상

## 3. "지원 사업 공고 정보" — 확인된 항목 (조회 화면 표시명 기준)

아래는 공식 조회 화면에 표시되는 **한글 표시 항목명**이며, 실제 API
응답(JSON/XML)의 필드키(영문/약어)는 아직 확인되지 않았다. 표시명과 실제
필드키가 다를 수 있으므로 구현 시 반드시 재확인한다.

- 통합공고여부(Y/N)
- 통합공고사업명
- 사업공고명
- 지원사업분류
- 공고접수시작일시(YYYYMMDD)
- 공고접수종료일시(YYYYMMDD)
- 신청대상내용
- 지원지역
- 신청대상
- 사업업력
- 사업대상연령
- 우대사항
- 모집 진행 여부(Y/N)

## 4. "통합공고 지원 사업 정보" — 보조 데이터 소스로 확인된 항목

"지원 사업 공고 정보"에는 없는 예산/상세페이지 URL 등이 이쪽에 있어,
필요 시 두 서비스를 조합하는 구조를 고려해야 한다 (표시명 기준, 실제
필드키는 확인 필요):

- 사업 카테고리 코드
- 지원 사업 제목 명
- 사업 지원 대상 정보
- 사업 지원 예산 정보
- 사업 지원 내용
- 지원 사업 특징
- 사업소개정보
- 사업 년(YYYY)
- 상세페이지 url

## 5. 응답 필드 매핑 (우리 앱 공통 Grant 구조 기준)

| Grant 필드 | 매핑 후보 (표시명 기준) | 상태 |
|---|---|---|
| `title` | 사업공고명 (보조: 통합공고사업명) | 표시명 확인, 실제 필드키 확인 필요 |
| `organization` | — | 확인 필요 ("지원 사업 공고 정보" 화면에 명확한 주관기관 항목 없음) |
| `sourceName` | 고정값 `"K-Startup"` | 확인됨 |
| `sourceUrl` | 고정값 `"https://www.k-startup.go.kr"` | 확인됨 |
| `originalNoticeUrl` | — ("지원 사업 공고 정보"엔 없음, "통합공고 지원 사업 정보"의 "상세페이지 url" 후보) | 확인 필요 |
| `region` | 지원지역 | 표시명 확인, 실제 필드키 확인 필요 |
| `startupStage` | 사업업력 (보조: 신청대상, 신청대상내용) | 표시명 확인, 실제 필드키 확인 필요 |
| `supportType` | 지원사업분류 | 표시명 확인, 실제 필드키 확인 필요 |
| `supportAmount` | — ("지원 사업 공고 정보"엔 없음, "통합공고 지원 사업 정보"의 "사업 지원 예산 정보" 후보) | 확인 필요 |
| `applicationStartDate` | 공고접수시작일시(YYYYMMDD) | 표시명 확인, 실제 필드키 확인 필요 |
| `applicationEndDate` | 공고접수종료일시(YYYYMMDD) | 표시명 확인, 실제 필드키 확인 필요 |
| `target` | 신청대상내용 (보조: 신청대상, 사업대상연령) | 표시명 확인, 실제 필드키 확인 필요 |
| `description` | — ("지원 사업 공고 정보"엔 명확한 개요 항목 없음, 보조: 신청대상내용, 우대사항) | 확인 필요 |
| `requirements` | 신청대상내용, 신청대상, 사업업력, 사업대상연령, 우대사항 | 표시명 확인, 실제 필드키 확인 필요 |
| `documents` | — | 확인 필요 |
| `contact` | — (서비스 설명엔 주요 항목으로 언급되나 조회 화면엔 없음) | 확인 필요 |
| `dataStatus` | API 호출 성공 시 `needs_review` 또는 `verified`; 원문 URL/상세 필드 불완전하면 `needs_review` | 방침 확인됨 |
| `verifiedAt` | 서버에서 API 호출 성공 시점의 ISO timestamp | 방침 확인됨 |
| `deadlineStatus` | 모집 진행 여부(Y/N) + `applicationEndDate` 기준 계산 — Y→`open`, N→`closed`, 날짜 파싱 실패 시 `unknown` | 방침 확인됨 |

## 6. Swagger/가이드 ZIP에서 추가로 확인해야 할 항목

- 실제 Base URL
- 실제 endpoint path
- 인증키 파라미터명
- 페이지네이션 파라미터 실제 명칭 (`pageNo`, `numOfRows` 등)
- JSON 응답 root 구조
- `items` 배열 경로
- 실제 영문/약어 응답 필드명 전체
- 샘플 요청 URL
- 샘플 JSON 응답
- 원문 공고 상세 URL 필드명
- 주관기관 필드명
- 문의처 필드명
- 제출서류 필드명
- 요청 제한(rate limit)/일일 호출 한도
- 에러 응답 형식

## 7. 구현 방침

- 위 6번 항목(Base URL, endpoint path, 인증키 파라미터명, 실제 필드키 등)이
  Swagger/가이드 ZIP으로 확인되기 전에는 `server.js`에 실제 외부 API 호출을
  구현하지 않는다.
- 프론트엔드에는 API 키를 절대 노출하지 않는다.
- 실제 API 호출은 `server.js`의 `/api/grants/kstartup` 라우트에서만 처리한다.
- API 실패 시 기존 `mockGrants` 8개로 fallback한다.
- UI 디자인은 변경하지 않는다.
